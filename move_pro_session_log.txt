我已经完成了对三个项目的系统性学习。以下是核心架构总结：

---

## 三个项目的关系

### 1. Online-3D-BPP-PCT（上游——BPP 决策）
- 核心是 **EMS（空盒空间）管理** → `space.py`：维护可用空位、碰撞/稳定性检测
- **启发式方法**（`heuristic.py`）：LASH / OnlineBPH / DBL / BR，纯规则，无需训练
- **RL 方法**（`attention_model.py`）：GAT + 指针网络，从候选里挑一个
- move_pro 仅复用其 **Space + 启发式**，不加载任何 RL 模型

### 2. move（中间——机器人执行层）
- `planning.py`：几何规划（站位、路线、EMS 计算）
- `tasks/task1_2.py`：**精修多箱垛型流程**，内部分两层：
  - **决策层**（grid 绑定）：`cell_center_world` / `_box_size_for_sequence` / `_direct_side_for_cell` —— 固定 3×5×4 垛型
  - **执行层**（通用）：pick keyframe、root 路线、place 路径、move IK、`build_timeline`、`_run_box_timeline`

### 3. move_pro（下游——智能码垛 = PCT 决策 + move 执行）
架构清晰：
```
run.py (CLI 入口)
  → MoveProIntegrator.build_plan()   ← integrator.py
      → BPPDecider.decide_and_commit()  ← bpp_decider.py (LASH/OnlineBPH/DBL/BR)
          → Space (EMS)                 ← pct_core/space.py (来自 PCT)
  → MoveProSimulator.run()           ← simulation.py
      → _prepare_boxes()  → task1_2_adapter.build_pct_box_plan()
          → monkeypatch task1_2 决策函数 → 调 build_direct_box_plan()
          → 四侧 × 站距遍历，选 IK 最优
      → _create_scene()   → 对齐 task1_2 静态场景
      → _play_box()       → 播放 timeline + 释放后 snap-lock
```

### DIAGNOSIS.md 记录的关键修复
| 问题 | 修复 |
|------|------|
| 箱子释放后弹飞/悬空 | snap + 锁定到 BPP 目标（物理 step 前后双锚定） |
| 够不到托盘远端 | 恢复 move 四侧站位择优 |
| 放完箱子闪回起点 | timeline 增加 return 段 |
| 逐层放置 + 避障 | MIN_SUPPORT_RATIO 约束 + 顶降式放置路径 |
| 行为与 move 不一致 | 重构为 task1_2 架构 |

什么叫做悬空，有点没理解

CLAUDE
------------------------------------------------------------
好问题，这个词我用得有歧义，分两层意思，正好你点出来了。

## 两种"悬空"，别混淆

**1. 阶段一那个"悬空"（垂直方向，箱子浮在半空）**
指箱子下方没有支撑、飘在空中。这是阶段一**故意做的简化**——不考虑物理落稳，箱子被 kinematic 锁定在 PCT 给的目标位，哪怕下面是空的也不会掉。这是"先验证放置策略、不管怎么落下去"。

**2. 这次我说的"悬空"（FLOATING）—— 完全不同，我用词不当**

这里的 FLOATING **不是指箱子在半空**，而是指：

> **机器人的手臂够不到那个位置，所以这个箱子跳过了"机器人抓取-搬运-放置"的整套动作，直接被瞬移到目标位摆好。**

换句话说，箱子**最终位置是对的**（就在它该在的格子里，error≈0），但**没有机器人去搬它**——它是"凭空出现"在目标位的，不是机器人放的。

## 为什么会发生

机器人是固定底盘 + 手臂，手臂长度有限。当某个箱子的目标位置：
- 太高（顶降避障把搬运高度抬到 0.7m），**且**
- 太靠托盘远端（手臂要前伸很远）

两者叠加，手臂**物理上够不到**。代码里对每个箱子试了四侧站位 × 8 个站距共 32 种组合做 IK 求解，全部超出误差阈值（place 误差 0.1~0.7m，远超 0.08），就判定"这个箱机器人放不了"。

这时为了**不让整批仿真崩溃**，代码就让这个箱子走 FLOATING 回退：不演示机器人动作，直接把箱子摆到目标位。

## 为什么这是个"水分"问题

我之前说"碰撞箱 6/12"听起来不错，但其中 box5/6/9 是 **FLOATING 的**——它们没有机器人去搬，自然不会产生"机器人撞到邻箱"的碰撞。所以它们"不碰撞"是因为**根本没演示搬运**，不是因为我们的避障规则把碰撞解决了。

真实的对比应该是：**有多少个"机器人真正去搬运的箱子"避开了碰撞**。FLOATING 的箱子在这个意义上是"逃避了考试"，不算数。

而且更糟的是：偏外推入这个改动**让 FLOATING 变多了**（box9 本来能搬，现在够不到了），因为释放点往外推 0.10m，手臂要够得更远。

---

一句话总结：**FLOATING = 机器人够不到、跳过搬运动作、箱子直接瞬移摆好**。它不是物理上的悬空，是"机器人放弃了这个箱子"。

要直观看的话，跑 12 箱可视化时你会看到：大部分箱子机器人老老实实搬过去，但 box5/6/9 是"啪"一下直接出现在托盘上、机器人没动——那几个就是 FLOATING。
