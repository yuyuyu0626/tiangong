## 文档结构（13 节）

### 1. 背景与动机

- 任务目标
- 输入和输出
- 核心困难
- 为什么需要当前设计
- 代码中哪些模块对应这些动机
- 与已有方法的关系（如适用）

### 2. 整体架构概览

说明代码仓库核心结构：
- 核心模型类
- 主要子模块及其职责
- 训练路径 vs 推理路径差异
- 数据从输入到输出的主要阶段
- 训练专用 vs 推理专用模块
- 参数共享与状态/缓存维护

给出完整流程示意，如：输入 → 编码 → 变换 → 上下文建模 → 概率估计 → 解码

### 3. 输入设定与符号定义

定义文档使用的所有符号，例如：
- `$B$` = batch size
- `$C$` = 通道数
- `$H, W$` = 空间分辨率
- `$N$` = token 数

每个变量须写明：语义、索引含义（如 `$X[b,c,h,w]$` 每个维度代表什么）、来源模块、后续使用模块。同一变量名不同阶段有不同形状时，分阶段说明。

### 4. 核心方法流程（重点）

按代码实际执行顺序，详细拆解核心算法。每个模块使用如下子结构：

#### 4.x.1 代码位置
文件、类、函数

#### 4.x.2 输入与输出

#### 4.x.3 具体计算过程
关键步骤转化为数学表达 + 自然语言。不跳步，局部变量与拼接后变量分别说明。

#### 4.x.4 变量语义解释
形状 + 含义，如 `attn[b,h,i,j]` = 第 b 样本、第 h head 中第 i 个 query 对第 j 个 key 的注意力权重

#### 4.x.5 设计动机
为什么这样实现，如为什么缓存、为什么 reshape、为什么训练推理不同

### 7. 训练流程与推理流程

#### 7.1 训练阶段
- 调用入口、输入构造、forward 路径
- loss 计算、梯度传播变量

#### 7.2 推理/压缩/解码阶段
- 调用入口、是否逐步处理
- 缓存/状态更新/上下文插入

### 8. 重要函数逐行级逻辑解释

选择最核心函数，包含：签名、参数、返回值、关键步骤、每步张量形状、语义、在流程中的作用、调用方。长函数可按逻辑拆分解释。

### 9. 模块之间的关系

解释不同模块如何配合：编码器→上下文模块→概率估计→熵编码→解码器，缓存如何服务 attention/context，mask 如何保证约束等。

### 10. 与已有方法或常见结构的关系

如借鉴了 Transformer、Swin、ViT、entropy bottleneck、hyperprior、autoregressive context、checkerboard context 等，说明：相似处、不同处、改动、改动目的、对复杂度/能力/效率的影响。无关联则不强行关联。

### 11. 复杂度与效率分析

分析核心模块计算复杂度和显存开销：attention token 数、window size、channel 数、head 数、slice 数、重复计算、缓存复用、推理/训练瓶颈。可用符号形式如 $O(B \cdot H \cdot W \cdot C^2)$，但须解释每个符号。

### 12. 容易误解或需要特别注意的地方

列出读代码时容易误解的点，如：
- 变量名看似单步但实际是多步拼接
- reshape 后维度顺序变化
- 函数名像训练用但推理也调用
- 缓存在插入阶段计算而非推理阶段
- mask 语义与常规不同
- 维度在不同阶段含义不同
- 训练并行 vs 推理逐步

### 13. 总结

回答：核心技术点、主要数据流、设计原因、最关键模块、决定效率/效果的实现细节、理解主线。

## 写作要求细节

### 公式格式
- 行内：`$公式$`
- 独立：`$$公式$$`
- **禁止** `\(...\)` 或 `\[...\]`

### 维度说明要求
每个变量必须包含：
1. 形状（如 `$B \times C \times H \times W$`）
2. 每个维度的语义
3. 索引含义（如 `$X[b,c,h,w]$` 中 b=样本, c=通道, h=行, w=列）
4. 来源模块
5. 后续用途

### 操作说明要求
遇到以下操作时必须展开：
- reshape/permute/view/transpose：说明前后维度、语义是否变化
- concat/split：说明拼接/拆分了什么、结果形状
- mask：说明 mask 语义（0/−∞/True/False 各代表什么）
- 缓存读写：说明存储内容、写入时机、读取时机、形状变化
- scatter/gather/unfold：说明索引语义、操作结果

### 准确性要求
- 严格基于代码分析，不编造
- 可确认的结论指出对应文件和函数
- 代码不够明确时写："代码中未能完全确认，可能需要结合训练脚本/配置/论文进一步确认"
- 代码与文档/注释/论文不一致时，以代码为准并指出差异

### 写作风格
- 像技术方案说明或论文方法部分扩展版
- 禁止笼统写法（"这里进行 attention 计算"、"然后 reshape"、"用于提取特征"、"推理时使用缓存"）
- 必须具体：Q/K/V 来自哪里、矩阵形状、元素含义、reshape 前后维度、缓存内容/读写时机/复用原因
